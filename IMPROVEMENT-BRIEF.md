# iscrt Improvement Brief: Blind Secret Provider for AI-Agent Workflows

## Purpose

`iscrt` is today a fine *interactive* secret manager: you `push`/`pull` `.env` files and `get` individual values as plaintext. It is **not** yet fit to be a *blind secret provider* — a tool you can hand to an autonomous coding agent (Claude Code) so the agent can run builds and deploys that NEED secrets without the agent ever being ABLE to read those secrets, and without plaintext ever touching the agent's context, terminal scrollback, or a committed file. This brief specifies the work to close that gap: a `run` (exec/inject) subcommand that injects secrets into a child process only, a safer story for `env pull`, and an optional "locked/agent mode" that disables reveal paths while still allowing injection. It is grounded in the current code; file/function references are concrete so a fresh agent can start implementing immediately.

---

## Current state (grounded in code)

- **Module:** `github.com/imaan/iscrt`, Go 1.26 (`go.mod`). Cobra + Viper CLI. Entry point `main.go` → `cmd.RootCmd.Execute()`.
- **Crypto** (`store/crypto.go`): Argon2id KDF (`argon2.IDKey(password, salt, 1, 64*1024, 4, 32)`) → **AES-256-GCM** (`aes.NewCipher` + `cipher.NewGCM`). Salt (16 bytes) + nonce prepended to ciphertext. *(See "Contradictions" — it is AES-GCM, not ChaCha20-Poly1305.)*
- **Store** (`store/store.go`): in-memory `map[string][]byte`, with `Get/Set/Has/List/Unset` plus prefix variants `ListPrefixContext` / `UnsetPrefixContext`. Project namespacing is purely a key-prefix convention: `project/KEY`.
- **Backends** (`backend/`): `local` (default, file at `SCRT_LOCAL_PATH`, written `0o600` in `local.go:130`), `s3`, `git`. Selected via `Backends[storage]` registry.
- **Unlock path** (`cmd/root.go` `PersistentPreRunE`): master password resolved by Viper from `SCRT_PASSWORD` env (prefix `scrt`, `AutomaticEnv`, key `password`) or `-p/--password`. If empty → hard error `"missing password"`. **There is no interactive TTY prompt** (verified: no `term.ReadPassword`/prompt anywhere). Password is excluded from verbose config logging (`root.go:116`).
- **Command surface** (`cmd/root.go` `init()` + `cmd/env.go` `init()`): `init`, `set`, `get`, `list`, `unset`, `storage`, and `env push|pull|list|delete`. **No `run`/`exec`/`inject` command exists** (verified).
- **The leak surfaces today:**
  - `cmd/get.go` decrypts one key and **prints plaintext to stdout** (`fmt.Println`/`fmt.Print`, TTY-aware via `isatty`). The only injection pattern available is shell substitution `VAR="$(iscrt get KEY)" cmd` — fragile: the value lands in the calling process's argv/env and any error/build log that echoes env leaks it; a bare `iscrt get` dumps plaintext into the agent's captured output.
  - `cmd/env.go` `envPull` → `cmd/envfile.go` `writeEnvFile` writes a **plaintext `.env` to disk** (mode `0600`, with a `# Generated by iscrt` header). Refuses to overwrite without `--force`. It does **not** check `.gitignore`. In a repo that must stay private this is a standing leak risk.
  - `env list -r` (`maskValue` with `reveal=true`) and `env list`/`get` all reveal plaintext on demand. Nothing distinguishes a human operator from an autonomous agent.
- **Project resolution** (`cmd/env.go` `gitRepoName`, lines 149-159): `git rev-parse --show-toplevel` → `filepath.Base`. So project defaults to the **git repo name**, not the cwd. (Flag help strings still say "current directory name" — stale.)
- **Test conventions:** `*_test.go` everywhere; table-ish unit tests using `github.com/golang/mock/gomock` against the `Backend` interface (`cmd/mock_backend.go`, regenerated via the `//go:generate mockgen` directive in `cmd/get_test.go`). Tests register a `"mock"` backend (`newMockFactory`), `viper.Reset()` + `viper.Set(configKey*)`, build a real encrypted blob with `store.WriteStore`, and drive `cmd.RunE` directly. Stdout is captured via `hijack()`/`restore()`/`hijackStdout` (see `cmd/main_test.go`). **Match this exactly** for new tests. `e2e/` holds end-to-end tests.

## Gap analysis (vs. the blind-provider purpose)

| # | Need | Today | Gap |
|---|------|-------|-----|
| 1 | Inject secrets into a child process, never printing them | only `iscrt get` (prints) + shell `$(...)` | **No `run` subcommand.** No way to pass secrets to a child without them transiting stdout/argv/the agent's context. |
| 2 | Keep plaintext off disk / out of git | `env pull` writes plaintext `.env`, no gitignore guard | **No in-memory alternative; no gitignore safety.** `run` (gap 1) is itself the fix; pull needs guardrails. |
| 3 | Safe to hand to an autonomous agent | every command can reveal plaintext | **No least-privilege mode.** Can't allow `run` while blocking `get`/`env pull`/`list -r`. |
| 4 | Don't leak via the tool's own output | `-v` logs config (password redacted) but child stdout/stderr is uncontrolled | Minor: must ensure `run` never logs values; optional value-redaction of child output. |

Non-gaps (already fine): encryption at rest, `0600` file perms, password kept out of verbose logs, project namespacing.

## Proposed changes

### A. `iscrt run` — inject-and-exec (the MVP)

**Signature**

```
iscrt run [--project NAME] [--only KEY1,KEY2,...] [--except KEY1,...] \
          [--no-inherit] [--require KEY1,...] -- <command> [args...]
```

**Behavior**

1. Resolve `project`: `--project` if set, else `gitRepoName()` (reuse the existing helper in `cmd/env.go`; do not re-implement). Error with the same message style as `env push` if neither is available.
2. Open backend, load + decrypt store exactly as `env.go` helpers do (`backend.Backends[storage].NewContext` → `ExistsContext` → `LoadContext` → `store.ReadStoreContext`). Factor the load-decrypt boilerplate that is currently copy-pasted across `get.go`/`env.go` into one helper (e.g. `openStore(cmd) (store.Store, error)`) and reuse it.
3. Collect keys under `project/` via `s.ListPrefixContext(prefix)`; strip the prefix to get env var names. Apply `--only` (allowlist) / `--except` (denylist) filtering. `--require` fails fast if any named key is missing.
4. Build the child environment:
   - Default: inherit parent env (`os.Environ()`) and overlay the decrypted secrets (secret wins on collision).
   - **Always strip `SCRT_PASSWORD` from the child environment** (in both inherit and `--no-inherit` modes). The master password decrypts *every project* in the store; the child needs this project's secrets, not the key to all of them. Without this, `iscrt run -- <anything>` hands the whole store to every process the child spawns. One line, mandatory, test-covered.
   - `--no-inherit`: start from an empty env, inject only the secrets (plus a minimal `PATH` so the command resolves — document this).
5. Exec the child with `os/exec`:
   - `c := exec.Command(args[0], args[1:]...)`; `c.Env = built`; `c.Stdin = os.Stdin`; `c.Stdout = os.Stdout`; `c.Stderr = os.Stderr`.
   - **Never** print, log, or echo any value or the assembled env. Do not run under `-v` logging that could dump it (the secrets must not pass through `logger`).
   - **Forward signals to the child.** `signal.Notify` for `SIGINT`/`SIGTERM` (and `SIGHUP`/`SIGQUIT`) in the parent, relay via `c.Process.Signal(sig)`. Without this, Ctrl-C on a long-running child (dev server, deploy) kills iscrt and orphans or double-signals the child. *Alternative considered and rejected:* `syscall.Exec` (replace the iscrt process entirely — simpler, no forwarding or exit-code plumbing, and wipes iscrt's memory holding the decrypted store). Rejected because it's untestable with the repo's `cmd.RunE`-driven test convention (it would replace the test process) and is Unix-only. Revisit if a hardened mode is ever needed.
   - Run; on exit, **propagate the child's exit code**. Extract it from `*exec.ExitError` (`exitErr.ExitCode()`); return it up so `main.go` exits with that status. Note: `main.go:handleError` currently maps generic errors to `os.Exit(-1)` — which the OS wraps to **255** — and only special-cases `syscall.Errno`. Add a typed error (e.g. `type exitCodeError struct{ code int }`) and handle it in `main.go` so `iscrt run -- false` exits `1`, not `255`. Set `SilenceErrors` for this error type (or print nothing) so the child's failure isn't wrapped in a noisy `Error: exit status N` line.
6. Use `--` as the argument separator (cobra: set `cmd.Flags().SetInterspersed(false)` or rely on `ArgsLenAtDash()` so flags after `--` go to the child). Validate that at least one command arg is present after `--`.

**Why this is leak-safe (and the honest limit):** secrets exist only in this process's memory and the child's environment block; they never hit stdout, never hit argv (so they don't show in `ps`/shell history the way `$(...)` expansions or `-e KEY=val` would), and never hit disk. **But the agent chooses the child command** — `iscrt run -- printenv` reveals everything. `run` is *accident-proofing*, not blindness: it removes the paths where secrets leak as a side effect of normal work. Actual enforcement against a curious agent lives in the *harness's* permission rules (e.g. Claude Code allow `iscrt run -- npm *` / deny bare `iscrt run -- sh`), which the README should spell out. True blindness would require the master password itself to be out of the agent's reach (see Risks — keychain endgame).

**Usage from an agent:** `iscrt run -- npm run build`, `iscrt run --only DATABASE_URL -- npx prisma migrate deploy`, `iscrt run --project myapp -- ./deploy.sh`.

### B. Make `env pull` safer

`env pull` writing plaintext to disk is the second leak surface. Options, in order of preference:

1. **`--check-gitignore` guard (recommended, default-on).** Before writing, verify the target file (default `.env`) is ignored by git (`git check-ignore -q <file>`; if it returns non-zero / the file isn't ignored, refuse with a clear error and tell the user to add it to `.gitignore` or pass `--allow-untracked`). Cheap, high value, matches the "repo must stay private" constraint. Skip the check gracefully when not in a git repo.
2. **`--stdout` mode.** Stream the `.env` body to stdout instead of a file, clearly intended for piping (`iscrt env pull --stdout > /dev/somewhere`). Lower priority — it reintroduces the stdout-leak risk for agents, so document it as "human/CI use, not for agents."
3. **Documentation nudge.** Update README to steer users toward `iscrt run` for the agent use case and frame `env pull` as the human/local-dev convenience.

Keep the existing `--force` overwrite behavior. The gitignore guard is on by default with an escape hatch flag — note that **default-on is a behavior break** for existing `env pull` scripts whose `.env` isn't gitignored; call it out as breaking in the changelog, don't bury it as "additive."

### C. Optional: agent/locked mode (least privilege for an LLM)

Add a mode that **disables reveal paths while still allowing injection**, so the binary is safe to expose to an autonomous agent.

- **Mechanism:** an env var `SCRT_LOCKED=1` (and/or a `--locked` persistent flag) checked in `RootCmd.PersistentPreRunE` *after* config load. **Naming matters:** `root.go:202` sets `viper.SetEnvPrefix("scrt")`, so Viper's `AutomaticEnv` reads `SCRT_*` — an `ISCRT_LOCKED` var would silently never trigger. Use `SCRT_LOCKED` (bound via Viper as key `locked`), and make at least one test set the *actual env var* (not `viper.Set`) so the binding is exercised. When locked:
  - Allow: `run`, `env push`, `env list` (masked, no `-r`), `list`, `storage`, `init`.
  - Block: `get`, `env pull`, and `env list -r` / any `--reveal`. Return a clear error like `"get is disabled in locked mode (SCRT_LOCKED); use 'iscrt run -- <cmd>' to inject secrets"`.
  - **Scope decision — reveals only, not writes.** Locked mode does *not* block destructive ops (`set`, `unset`, `env delete --force`, `env push --mode replace`): an agent in locked mode can still wipe or overwrite the store. Blocking writes would break the legitimate agent workflow (`env push` after adding a key). Document this explicitly in the README so nobody mistakes locked mode for read-*and*-write protection; backups/versioned backends are the mitigation for destructive mistakes.
- Implement by tagging commands (e.g. an annotation `cmd.Annotations["reveals_plaintext"]="true"` on `getCmd` and a check on `--reveal` for `env list`) and short-circuiting in `PersistentPreRunE` based on the resolved command + flags.
- **Tradeoffs (state these in code comments/README):** This is a *guardrail, not a security boundary*. Anyone who can set `ISCRT_LOCKED` can unset it, and anyone with the master password + the store file can decrypt out-of-band. Its value is preventing *accidental* reveal by an agent that's only been told to run builds — it removes the easy `iscrt get` footgun from the agent's toolbelt. Worth doing; do not oversell it.

### D. Optional / note-only: redact secret values from child stdout/stderr

`op run` masks known secret values appearing in the child's output (replacing them with `<concealed by op>`). To do this `iscrt run` would wrap `c.Stdout`/`c.Stderr` in a streaming filter that scans for any injected value and replaces it.

**Recommendation: defer / probably skip.** It adds real complexity (streaming buffered matching across chunk boundaries, performance on large build logs, false sense of safety) and does not address the core threat — a build that writes its own env to a file or ships it in a bundle. Note it as a possible Phase 4 if a concrete need appears. The honest framing: env-var injection inherently exposes the value to the child and everything it spawns; masking only cleans up *display*, not propagation.

### Phase 1 — `iscrt run` (MVP) + exit-code propagation
**Build:** the `run` command (section A). Factor a shared `openStore` helper.
**Files to touch:**
- `cmd/run.go` (new) — the command, flag parsing, env assembly, exec.
- `cmd/root.go` — `addCommand(runCmd)` in `init()`.
- `cmd/store_helpers.go` (new) or top of `run.go` — extract the load+decrypt boilerplate; optionally refactor `get.go`/`env.go` to use it (low risk, but can be deferred to keep the diff tight).
- `main.go` — handle the typed exit-code error so the child's status propagates.
**Tests to add (`cmd/run_test.go`):** mirror `get_test.go` conventions (gomock `Backend`, `viper.Reset()`, build store with `store.WriteStore`). Cases:
- injects project keys into a child (use a harmless command like `sh -c 'echo "$MY_KEY"'` captured via the `hijack()` helper, asserting the value reaches the *child*, then assert iscrt's own non-child output never contains it);
- **`SCRT_PASSWORD` is stripped from the child env** (set it in the parent, `sh -c 'echo "${SCRT_PASSWORD:-ABSENT}"'` → `ABSENT`) — this is the security-critical test, in both inherit and `--no-inherit` modes;
- `--only` / `--except` filtering; `--require` missing key → error;
- `--no-inherit` excludes parent env;
- exit code propagation (`-- sh -c 'exit 7'` → error carrying code 7);
- missing project / not-a-git-repo error path;
- no command after `--` → usage error;
- **child flags survive the root re-parse**: `PersistentPreRunE` re-parses `os.Args[1:]` after adding backend flags (`root.go:102`), and `addCommand` sets `UnknownFlags = true` — test that `iscrt run -- some-cmd -p foo --storage x` passes `-p`/`--storage` to the child untouched.

These tests spawn a real `sh`, which makes `cmd/` tests Unix-only where they currently aren't. Accept that constraint explicitly (the tool targets macOS/Linux); guard with `runtime.GOOS != "windows"` skips rather than adding a helper-process indirection.

## Risks & non-goals

- **The agent controls the child command.** `iscrt run -- printenv` (or `-- sh -c 'echo $DATABASE_URL'`) reveals everything injected. Locked mode cannot close this — it would mean allowlisting child commands, which is the *harness's* job (Claude Code permission rules), not iscrt's. Frame everything in this brief as accident-proofing for a well-behaved agent.
- **The master password is reachable by the agent, so "blind" has a ceiling.** For `run` to work non-interactively, `SCRT_PASSWORD` must be in the agent's environment (or a `.scrt` config file — note `readConfig` searches *upward through parent dirs*, another read path). An agent holding the password + the store file can always decrypt out-of-band. The endgame for actual blindness is getting the password out of the agent's reach entirely — e.g. macOS Keychain / OS keyring integration where iscrt fetches the password itself and no env var ever exists, combined with locked mode. Out of scope for these phases; record it as the strategic direction.
- **Env-var injection still exposes secrets to the child and everything it spawns.** `iscrt run` cannot stop a build script that prints its own environment, writes `.env` to disk, or bakes secrets into a client bundle. This is inherent to passing secrets to a process; document it plainly. (Mitigated one level: `SCRT_PASSWORD` itself is stripped from the child env, so a leaky child exposes this project's secrets, not the whole store.)
- **Client-side bundles leak regardless.** Any secret needed in browser-shipped JS is public once shipped. `run` doesn't change that — it's a build/deploy/server-side tool.
- **Locked mode is a guardrail, not a sandbox.** It prevents accidental reveal by a well-behaved agent; it does not defend against a malicious process with the master password or read access to the store file. Don't market it as isolation.
- **`--no-inherit` can break tools** that need `HOME`, `PATH`, etc. Inject a minimal safe set or document that users must pass needed vars through. Default stays inherit-and-overlay.
- **Process argv is visible; environment is much less so but not secret from the same user.** `run` avoids the argv leak of `-e KEY=val`-style tools, but a user who can `ps -E` / read `/proc/<pid>/environ` as the same uid can still see the child's env. This is strictly better than `$(iscrt get)` in shell history/logs, not a hardware boundary.
- **Non-goal:** changing the crypto, the store format, or backend selection. Leave AES-256-GCM/Argon2id and the `local`/`s3`/`git` backends untouched.

### Contradictions found vs. the working assumptions I was handed

- **Cipher is AES-256-GCM, not ChaCha20-Poly1305.** `store/crypto.go` uses `aes.NewCipher` + `cipher.NewGCM`; README confirms "AES-256-GCM". KDF is Argon2id, as stated.
- **There is no interactive password prompt.** Password comes from `SCRT_PASSWORD` or `-p/--password` only; if both are empty, `PersistentPreRunE` returns `"missing password"` and exits. The "or prompt" fallback does not exist in code.
- **Project default is the git repo name, not the cwd directory name.** `gitRepoName()` uses `git rev-parse --show-toplevel` + `filepath.Base`. The flag help text ("current directory name") is stale and worth fixing.

Everything else in the working assumptions checks out: `~/.iscrt/store.iscrt` written `0600`, command surface, plaintext-on-`get`, plaintext `.env` on `pull`, no `run`/`exec`/`inject` command, three backends.
